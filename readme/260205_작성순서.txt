========================================
실무에서 공통 CSS 만드는 타이밍
========================================

1. 절대 시작하자마자 만들지 않는다
- 아직 화면이 몇 개 나올지 모른다
- 디자인 방향도 안 잡혀 있다
- 공통으로 묶을 기준이 없다

2. 최소 2~3개 화면이 나온 뒤 만든다
- Login
- Tasks
- TaskDetail

이 정도 화면이 나오면
버튼, input, margin, layout 패턴이 보이기 시작한다

3. "이거 계속 똑같네?" 라는 순간이 기준
예시
- 모든 페이지 버튼이 거의 같은 스타일
- 페이지마다 padding, width가 거의 동일
- h1, h2 스타일이 반복됨

이때 공통 CSS를 만든다

========================================
공통 버튼 컴포넌트 만드는 타이밍
========================================

조건
- 같은 버튼 스타일이 두 번 이상 등장
- props로 제어할 수 있을 때

만들기 전 질문
- 이 버튼은 역할이 같은가?
- 스타일만 같은 게 아니라 의미도 같은가?

예시 (만들어도 되는 경우)
- Login 버튼
- Add 버튼
- Save 버튼

예시 (아직 만들면 안 되는 경우)
- 삭제 버튼
- 위험 버튼
- 강조 버튼

"스타일이 같아 보여도
의미가 다르면
절대 일찍 공통화하지 않는다"

이유
- 역할이 다르면 나중에 분기 지옥이 된다

========================================
실무 기준 안전한 전략
========================================

초기
- 기본 Button만 만든다
- 의미 없는 UI 레벨 컴포넌트만 공통화

중반
- 패턴이 보이면 DangerButton, ConfirmDeleteButton 처럼
  "의미가 드러나는 컴포넌트"로 만든다

후반
- 디자인 시스템 단위로 통합한다

========================================
실무 기준 공통화 순서
========================================

1. 페이지 2~3개 먼저 만든다
2. 중복되는 스타일을 발견한다
3. 공통 CSS 파일 생성
4. 버튼, 인풋 같은 아주 기본 컴포넌트만 만든다
5. 페이지별 특수한 UI는 절대 공통화하지 않는다
